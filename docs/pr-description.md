# fix(map-sync): restore native rendering and complete shared maps

## 修改目标

解决本机飞行地图被服务端结果覆盖、同步地图出现棋盘状空洞和颜色失真、客户端退出后地图随机残缺，以及服务端处理慢或失败后无法可靠补偿的问题。

## 修改内容

- 服务端在瓦片原子落盘并发布新 revision 后主动向同维度客户端推送，Merkle 轮询保留为断线和背压恢复机制。
- 地图压缩与落盘从服务器 tick 解耦，使用 2 至 4 条按坐标有序的 writer；候选扫描默认 512，渲染受 16 块、25ms 和实时 tick 余量约束。
- dirty 任务只在瓦片落盘和索引发布成功后确认；失败任务保留并退避重试，本机完成提示可把仍缺失的区块提升到队列前部。
- 客户端持久化完整瓦片内容，不再把“已注入内存的 revision”误当成“Xaero 已保存到磁盘”。
- 每次连接清空会话应用状态，通过服务端 revision 与 content hash 校验本地瓦片；命中时直接恢复，缺失或损坏时才请求服务端。
- Xaero 对当前已加载区块生成的本机地图始终优先；服务端瓦片只补本机未加载区域，不覆盖正在生成或已经完成的本机结果。
- 缓存瓦片按 Xaero 32x32 chunk 区域批量应用，每批只刷新一次区域，并在失败时整批回滚。
- 客户端限流扫描附近的 Xaero 已完成区块，只向服务端发送坐标和已有 hash 提示；不上传地图画面，服务端仍生成权威瓦片。
- v5 地图格式使用稳定生物群系键并保留 Xaero 透明覆盖的累积不透明度，修复动态注册表颜色错位和水体/玻璃覆盖失真。
- Xaero `loadState=0/4` 即使残留 reload 标记也会重新进入原生加载队列，避免重启恢复永久卡住。
- Merkle 根和节点请求增加同步代次与请求编号，迟到、重复或跨连接响应不会推进当前同步；只有本地叶子重新计算出的根匹配才算完成。
- 客户端与服务端在断线时清理分片传输会话，防止旧连接数据污染新会话。
- 协议和地图格式升级到 5，预发布版本升级到 `2.1.0-beta.11`。

## 不包含的内容

- 不强制加载历史 Minecraft 区块，不跨线程读取世界对象。
- 不修改 Xaero 原生地图文件格式或公开路径点权限模型。
- 不删除旧缓存和旧协议数据文件。

## 风险等级

Medium

## 行为变化

修改前：客户端只持久化 revision；若 Xaero 尚未完成原生保存便退出，重进后会跳过实际缺失的瓦片，并持续向服务端轮询。

修改后：客户端本机生成结果优先，服务端批量补齐未加载区域；完整瓦片缓存可校验恢复，服务端主动推送变化，断线后再由带代次的 Merkle 补偿。

## 评测结果

基线结果：首次同步约 4027 个服务端瓦片请求；旧实现重启后只保留 4028 条 revision，Xaero 原生区域文件不完整时会出现随机空洞。

修改后结果：179 项自动化测试通过；本地命中不请求瓦片 body，远程结果按区域批量应用，旧 Merkle 响应和旧 revision 不再覆盖新状态。`beta.11` 尚未在当前仍运行旧 JAR 的双客户端中完成重启视觉验证。

差异：本地缓存命中消除了重复瓦片下载；首次连接或服务端内容变化仍按需传输。

## 性能和成本

响应时间：新 revision 发布后主动推送；重启恢复主要受本地磁盘读取和 Xaero 原生注入速度限制。

Token 消耗：不涉及模型调用。

工具调用次数：本地命中不产生瓦片数据请求；Merkle 根校验仍按恢复策略执行。

## 安全影响

是否增加权限：否。

是否增加数据访问：客户端仅缓存原本有权通过共享地图协议接收的瓦片。

是否增加网络访问：不增加外部网络访问。

是否涉及敏感信息：否。

## 发布方式

预发布版本 `2.1.0-beta.11`，服务端和客户端必须同步灰度升级，并观察同步耗时、persistent holes、磁盘占用、网络流量与 MSPT。

## 回滚方式

停止服务端并将服务端与所有客户端 JAR 回滚到提交前版本。协议 5 客户端与服务端必须同步升级或同步回滚；`tiles-v5`、`map_tile_index-v5.json` 和客户端 v5 缓存可以保留，旧版本不会读取。
